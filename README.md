# Genie — Personal Assistant Slack Bot

Genie is a Python-based personal assistant that lives in Slack. It responds to messages and mentions, pulls data from Gmail, Google Calendar, and Linear, uses an LLM to answer freeform questions, and delivers a morning briefing every day at 7:30 AM GMT automatically.

## Architecture Overview

```
┌─────────────────────────────────────────────────────────┐
│                        main.py                          │
│                                                         │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │ Slack Bolt  │  │ APScheduler  │  │  Flask /health│  │
│  │ (Socket     │  │ (7:30 AM     │  │  (port 8080)  │  │
│  │  Mode)      │  │  briefing)   │  │               │  │
│  └──────┬──────┘  └──────┬───────┘  └───────────────┘  │
│         │                │                              │
│         └────────┬───────┘                              │
│                  ▼                                       │
│            route(text, say)                             │
│                  │                                       │
│    ┌─────────────┼──────────────┐                       │
│    ▼             ▼              ▼                        │
│  Gmail      Calendar /      Linear /       AI           │
│  emails     briefing        issues      (fallback)      │
└─────────────────────────────────────────────────────────┘
         ▲
         │ outbound WebSocket (Socket Mode)
         │ — no inbound port required for Slack
         ▼
      Slack API
```

Genie connects **outward** to Slack via a persistent WebSocket (Socket Mode). Slack does not make HTTP calls into your server. This means no public URL or open inbound firewall rule is needed for Slack communication.

## How It All Fits Together

When `main.py` starts, four things happen concurrently:

1. **Flask** starts on port 8080 in a background daemon thread to serve the `/health` endpoint.
2. **APScheduler** starts a background job that calls `send_briefing()` every day at 7:30 AM (server time — must be GMT).
3. **Slack Bolt** registers two event listeners: one for direct messages (`@app.message`) and one for `@mentions` (`@app.event("app_mention")`).
4. **SocketModeHandler** opens an outbound WebSocket to Slack and blocks — this is the main thread keeping the process alive.

Every inbound Slack message (whether a DM or a mention) is passed through `route()`, which keyword-matches the text and delegates to the appropriate integration or falls back to the AI.

## Environment Variables

Create a `.env` file in the project root. **Never commit this file.**

```env
SLACK_BOT_TOKEN=xoxb-...        # Bot User OAuth Token — authorises Slack API calls
SLACK_APP_TOKEN=xapp-...        # App-Level Token — used only for Socket Mode WebSocket
SLACK_USER_ID=U...              # Your personal Slack user ID — briefing is DMed here
GROQ_API_KEY=gsk_...            # Groq API key for LLaMA inference
LINEAR_API_KEY=lin_api_...      # Linear personal API key
SENTRY_DSN=https://...          # Sentry DSN for error tracking
```

### Token types explained

| Token | Prefix | Purpose |
|---|---|---|
| Bot Token | `xoxb-` | OAuth bearer token — grants permission to call Slack APIs (post messages, read events). Tied to your bot's installed scopes. |
| App Token | `xapp-` | Transport token — used exclusively to open the Socket Mode WebSocket connection to Slack. Separate from API permissions. |

If either token is revoked, the bot fails: the bot token breaks API calls, the app token breaks the WebSocket connection entirely.

## Google OAuth Setup

Google auth is a two-step process: one manual step to generate a token, then automatic refresh from that point on.

### Step 1 — Get your OAuth client credentials

1. Go to [Google Cloud Console](https://console.cloud.google.com)
2. Create a project and enable **Gmail API** and **Google Calendar API**
3. Under *Credentials*, create an **OAuth 2.0 Client ID** (Desktop app type)
4. Download the JSON file and save it as `credentials.json` in the project root

### Step 2 — Run the authorisation flow (once only)

```bash
python authorise.py
```

This prints a browser URL. After you approve access, paste the authorisation code back into the terminal. It writes the token directly to `credentials/token.json`.

### Token refresh

`utils/google_auth.py` automatically refreshes the access token when it expires using the stored refresh token. If the token file is missing or the refresh fails, a clear error is raised with instructions to re-run `authorise.py`. You only need to re-authorise if the refresh token itself is revoked (e.g. you revoke access in your Google account security settings).

## Slack App Setup

1. Go to [api.slack.com/apps](https://api.slack.com/apps) and create a new app *From Scratch*
2. Under **OAuth & Permissions**, add these Bot Token Scopes:
   - `chat:write`
   - `im:history`
   - `im:read`
   - `app_mentions:read`
3. Under **Socket Mode**, enable it and generate an App-Level Token with the `connections:write` scope — this becomes your `SLACK_APP_TOKEN`
4. Under **Event Subscriptions**, enable events and subscribe to:
   - `message.im` — catches DMs to the bot
   - `app_mention` — catches @mentions in channels
5. Install the app to your workspace — this generates your `SLACK_BOT_TOKEN`
6. Find your personal Slack User ID (click your profile → *More* → copy Member ID) and set it as `SLACK_USER_ID`

## Receiving Messages

Genie listens for two Slack event types, both handled in `main.py`:

```python
@app.event("app_mention")   # User tags @Genie in a channel
def handle_mention(event, say): ...

@app.message()              # User sends Genie a direct message
def handle_message(message, say): ...
```

Both handlers lowercase the message text and pass it to `route()`.

**Why Socket Mode?**
Socket Mode opens an outbound WebSocket connection from the bot to Slack. Slack delivers events over that connection rather than making HTTP requests to your server. This means:
- No public URL required
- No inbound firewall rules required
- Works on a server behind NAT or inside a private network
- Trade-off: the process must stay running — if it crashes, no events are delivered until it restarts

## Routing Logic

`route()` in `main.py` is the decision layer. It immediately sends a thinking message, then keyword-matches the lowercased text to decide what to do:

```python
def route(text, say):
    say(THINKING_TEXT)   # Immediate acknowledgement to the user

    if "email" in text:
        result = get_unread_emails()
    elif "calendar" in text:
        result = get_todays_events()
    elif "linear" in text:
        result = get_pending_issues()
    elif "briefing" in text:
        result = get_briefing()
    elif "help" in text:
        say(HELP_TEXT)
        return
    else:
        result = get_ai_response(text)   # Freeform AI fallback
```

| Keyword in message | What happens |
|---|---|
| `email` | Fetches up to 5 unread Gmail messages |
| `calendar` | Fetches today's Google Calendar events |
| `linear` | Queries Linear GraphQL API for open issues |
| `briefing` | Generates a full morning briefing via AI |
| `help` | Returns the static help menu |
| anything else | Sent to Groq LLM for a freeform answer |

Routing is simple `if/elif` — first keyword match wins. If a message contains both "email" and "calendar", only emails are returned. This is intentional simplicity; extend with more specific matching if needed.

## The Morning Briefing

`send_briefing()` runs automatically at 7:30 AM server time every day via APScheduler.

```python
scheduler.add_job(send_briefing, "cron", hour=7, minute=30, timezone="GMT")
```

The briefing flow:

```
APScheduler (7:30 AM GMT)
    └── send_briefing()
            ├── get_briefing()
            │       ├── get_unread_emails()      → Gmail API
            │       ├── get_todays_events()      → Google Calendar API
            │       └── get_ai_response(prompt)  → Groq LLM
            └── app.client.chat_postMessage()    → DM to SLACK_USER_ID
```

`briefing.py` assembles a prompt containing live email and calendar data, then passes it to the LLM to write a warm, professional morning summary. The LLM is explicitly instructed to list all calendar events without skipping any, open with "Good Morning", and close with a motivating line.

**Timezone:** The scheduler explicitly uses `timezone="GMT"`, so the briefing fires at 7:30 AM GMT regardless of the server's local timezone.

## Integrations

### Gmail — `integrations/gmail.py`

- Authenticates via Google OAuth (read-only scope: `gmail.readonly`)
- Queries the Gmail API for up to 5 unread messages in the inbox
- Returns sender and subject line for each — does not read email body content

### Google Calendar — `integrations/calendar.py`

- Authenticates via Google OAuth (read-only scope: `calendar.readonly`)
- Queries the primary calendar for all events between midnight and 23:59 UTC today
- Returns event title and start time formatted as `H:MM AM/PM`

### Linear — `integrations/linear.py`

- Authenticates via a static API key (no OAuth flow required)
- Sends a GraphQL query to `https://api.linear.app/graphql`
- Returns all issues where state type is not `completed` or `cancelled`
- Each issue includes: title, state, priority label, assignee, URL, and due date
- Webhook signature verification and event formatting were removed — the bot uses on-demand polling only

## AI Layer

`ai_helper.py` wraps the Groq API:

```python
client = Groq(api_key=os.environ["GROQ_API_KEY"])

response = client.chat.completions.create(
    model="llama-3.3-70b-versatile",
    messages=[{"role": "user", "content": prompt}]
)
```

The model is **LLaMA 3.3 70B** served via Groq's inference API. It is used for:
- Freeform questions that do not match any keyword
- Composing the morning briefing narrative from raw email and calendar data

The active AI provider is Groq (LLaMA 3.3 70B). To switch providers, replace the `ai_helper` import in `main.py` and `briefing.py` with a compatible wrapper and update your `.env` accordingly.

## Error Tracking

Sentry is initialised at startup with full tracing. The DSN is read from `os.environ["SENTRY_DSN"]` — no hardcoded values:

```python
sentry_sdk.init(
    dsn=os.environ["SENTRY_DSN"],
    send_default_pii=False,
    traces_sample_rate=1.0,
    profile_session_sample_rate=1.0,
)
```

Exceptions are captured in two places:
- `route()` — any integration failure during message handling
- `send_briefing()` — any failure during the scheduled briefing

When an error occurs in `route()`, Genie sends the user a fallback error message rather than failing silently.

## Known Gaps

| Gap | Impact | Fix |
|---|---|---|
| No startup persistence / process manager | If the process crashes, no Slack events are delivered until it restarts | Use `systemd`, `pm2`, or a platform that auto-restarts on crash (Railway, Fly.io, Render) |
| Routing is first-match keyword only | A message with multiple keywords only triggers the first match | Extend `route()` with more specific matching or intent detection if needed |
